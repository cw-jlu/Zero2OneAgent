## 模块一：Context / Performance

### Q1 
大模型面对第一轮长窗口或多模态输入时，first token 会显著变慢。有什么快速/低成本/用户体验也不差的方案？从5-10秒稳定压缩到2秒。

### A1
1. 预填充+软硬件加速。一些（因为是底层算法相关）我不是很熟悉的（没有在工业场景下实操）的方案。众所周知现在的transformer需要做这些QKV的计算，使用prefill预填充向量值到显卡可以加速这一过程，另外还有使用一些矩阵运算的优化，比如使用矩阵乘法代替循环，使用并行计算等等，有一些推理加速的方法，比如做一些矩阵的切分等等，另外还有就是硬件层面有一些硬件的加速器，不过这些都是需要特定的开源模型才能做到的，比如Megatron-LM，DeepSpeed等等。我的第二个项目就是在A40显卡上做的服务部署，用的是Qwen-omni，当时做压测，并发应该是200左右除了模型推理这块其他所有接口都大概在300ms内，但是首字延迟有10s，后面就做了软件加速
2. 动态更换模型（flash版本或者量化模型）。量化模型因为计算简单了所以算得快，如果场景适配的话，成本不高，用户体验也差别不大（虽然损失了一部分性能），动态选择的策略可以是（长窗口的长度字符数/是否是多模态输入）

### Q2
一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

### A2
怎么做一个好的压缩呢？我们第一次想，可能想到的是，去掉一些太早的对话或者压缩一些太早的对话（比如说对比较早部分的对话用大模型进行摘要）。但是这样会有一些问题，就是如果用户问了一个跟比较久以前的对话相关的问题，那么这段记忆就丢失了，完全不知道用户问什么，另外多久算久？不同的场景下（首先在不同的使用场景编程等会有不一样，另外用户可能只是没有切换session，今天的对话和昨天的对话完全无关），带上之前的记忆不但会撑爆context，还会污染上下文，使回复质量变差。

所以，我们不能这么简单粗暴的干，我想到以下几点：
1. 首先，我们能做什么。去掉失败的loop，这在我的个人项目data agent也有使用，比如调用工具失败了几次，最后成功了，或者是因为网络问题造成的错误等等诸如此类问题，完全可以在成功后把这些失败的loop去掉。不要觉得留下来可以有一些memory类似的效果，减少后面犯同样的错误，实际上根据我的经验，这些失败的loop，下次再出现的时候，还是会犯同样的错误，出了类似问题，最应该检查的是rules，example这些全局prompt，memory，tool description有没有做好。
2. 去掉/压缩一些不那么必要的东西。context里面塞了很多东西，可能有rag召回的东西，tool/skill定义，schema定义，memory，these东西占了一部分，虽然不是很多，但是是否有冗余这是个可以改进的点。另外一部分，context的大头就是历史对话记录，像ReAct，比如设计要保存thought，action，observation，我们把较久的和现在对话时间上不连续的action删掉或压缩，比如说action是调用tools，那么我们可以只保留agent的输入和结果就好了，至于中间过程是什么调用什么tool我们不关心，对后面对话影响也不大。
3. 动态压缩context：比如说超过一个设定的上下文窗口的阈值（如xxtoken，75%context window），那么就动态压缩context，比如只保留最近的50%的context，前面用模型做压缩摘要。
4. 动态裁剪context（成本较高，效果可能不好，调试比较麻烦，一般我觉得前三个就够了，一般不会有用户的一个session里塞几百万字的）。对比前文对话记录，筛去/压缩和当前无关的

## 模块二：Memory

### Q1
和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

### A1
1. 精确匹配，如果匹配度较高，那么就可以认为是之前问过的问题，然后直接返回之前的问题的答案，或者是把这一历史信息加入上下文中，告诉模型这个之前什么时候问过一样的，前者成本低，快速，后者可能更能结合上下文，回复更懂用户。匹配办法大致有两种效果还行的，第一种是传统的BM25+cosine embedding计算，第二种是用另一个模型（比如一个低成本的小模型）来判断，效果一般来说不错。
2. 模糊/语义匹配及多级召回（Semantic Search & Multi-stage Retrieval）。对于无法精确匹配的复杂提问，利用 Vector Embedding 对历史记忆库进行向量相似度检索。为了提高准确度并节约上下文，可以采用多级召回架构：先通过语义检索（例如检索过去 15 天的交互摘要或关键实体）过滤候选集，再通过轻量 Reranker 重排筛选出置信度最高的相关对话片段，以“历史记忆背景”的方式注入当前 Context，让模型产生“认识用户、记得过去”的表现。

### Q2
你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

### A2
其实我最开始接触memory当时的做法还是简单的做一个召回，有点像对历史context做的一个rag，后来的memory多了一些遗忘机制还有一些world model之类的元素，不过这都是学术界的玩法，工业界基本就是多级多类型memory框架，也是Agent Memory的经典框架。

趋势：memory的遗忘，整合，重构，获取。

头部：我看过Hermes的memory设计，他们家做的比较有意思，我也认可他们的思路，即，less is more（少即是多）。他们把长期固定的少量全局memory放入prompt（用户画像这些），其他的一般memory作为tool去让llm自己决定要不要调用，也就是渐进式披露的思想，更少的memory效果反而一般更好，因为agent的memory跟人类的是不等价的

## 模块三：Task

### Q1
对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

### A1
1. 策略1：subagent
   使用subagent，主agent只专注于推进主体的任务，有些事情交给subagent，但是要注意subagent可能无法理解全局，会有偏差，但是设计的好这种机制效果还是不错的
2. 策略2：压缩
   **核心思想**：定期将冗长的上下文压缩为精炼的摘要，释放空间给新信息。
   这是最直觉的策略——就像整理书桌一样，定期把堆积的文件归档成摘要，腾出空间放新的工作材料。如果你在 7.2 节理解了摘要压缩技术，那么压缩整合可以看作它的"升级版"——不仅仅是压缩单段对话，而是系统性地管理整个对话历史的生命周期。
   Anthropic 的 Claude Agent 在实际产品中就使用了这种策略：当上下文接近窗口限制时，自动触发"compaction"（压缩整合），将历史对话压缩为结构化摘要。这个机制让 Claude Agent 能够处理远超窗口大小的长对话，而不会出现质量退化。
   但是压缩的问题正如上文所说，如何压缩，压缩到什么程度，压缩后如何保证流畅性，这些都是需要考虑的问题，压缩会丢失信息。
3. 策略3：结构化笔记（Markdown）
   **核心思想**：Agent 在执行过程中主动维护一份结构化的"笔记"（比如说plan.md/todo list），记录关键信息和任务状态，而不是依赖完整的对话历史。
   这个策略来源于人类在执行长期项目时，不会试图记住每一次对话的每一个字，而是维护一份笔记。当你需要回忆某个细节时，你不是翻看完整的会议录音，而是查看自己的笔记。Agent 也应该这样工作。
   这种方式比2更高效。因为由 Agent 自己提炼的质量远高于事后的压缩。在做决策的当下，Agent 最清楚哪些信息是重要的、哪些中间结果需要保留。事后再让一个独立的压缩模型来判断"什么重要"，难免会有偏差。

总结就是我觉得策略1和3是不错的，2最好是做为兜底的实在塞不下的策略

## 模块四：Tool / Session Runtime

### Q1
Agent 工工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

### A1
1. **异步任务生命周期与解耦执行**：
   运行时引入异步任务队列和后台 Worker。当 Agent 调用长耗时工具（例如“跑自动化测试集”、“构建代码”或“抓取大量网页”）时，Runtime 立即拦截该请求，并在进程/线程池中启动异步 Worker，同时向 Agent 返回一个包含任务 ID 的挂起状态回执（例如 `{"status": "pending", "task_id": "job_9527"}`）。
   Agent 接收到 pending 状态后，可以输出友好的中间进展提示（“我已在后台为您运行测试，可能需要 1 分钟左右，有结果时我会立即通知您”），然后结束当前轮次，将状态挂起或保持空闲。
2. **通知与回调模式**：
   **事件驱动机制**：异步 Worker 执行完成后，向 Runtime 事件总线投递一个完成事件，写入 Session 对应的 Event Queue。
   **双向通信实时推**：若用户正停留在 UI 界面上，Runtime 通过 WebSocket 或 SSE 向前端推送实时状态变更，使前端能够直观展示任务进度和最终结果。
3. **主动重入交互流（Re-entry）**：
   任务完成后，若 Agent 正处于等待状态，事件监听器自动触发一个新的“推理轮次 (Turn)”，将该异步工具的最终执行结果作为 `Observation`（附加 `async_job_id` 等上下文标识）塞入 Agent 的上下文。
   驱动 Agent 根据结果进行下一阶段的思考和总结，输出最终结果给用户。

### Q2
如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

### A2
在并发事件到达时，Runtime 通常采取“队列排队 + 信号中断”的机制：
1. **用户又发来新消息（抢占与中断）**：
   **如果是交互式/协作式 Agent（如终端助手 Claude Code）**：
     **强抢占/中断策略**：Runtime 捕捉到用户新的输入后，向当前的 LLM 生成进程或非关键的 Tool 线程发送 cancellation 信号（利用 cancellation token），强制中止当前生成。
     Runtime 自动保存当前的中间推理状态（Thought），在 Message History 中追加一条“【系统提示：生成被用户新消息中断】”，并将用户的新输入拼接到后面，立即开启一轮新的 Prompt 推理。
   **如果是任务导向/后台型 Agent（如自主执行 Bug 修复的机器人）**：
     **排队缓冲策略**：将用户的新消息放入 Session 队列（Buffer Queue），Session 维持 busy 状态继续运行，直至当前 Turn 执行完毕进入 idle，再弹入队列中的用户消息开始下一轮。
2. **异步工具完成事件到达**：
   Runtime 接收到完成事件后，不应该直接打断 Agent 正在进行的 LLM 文本生成，因为强行插入会破坏模型的 CoT 生成结构。
   **处理策略**：异步事件被追加到 Session 的“待消费事件队列 (Pending Events)”。当 Agent 完成当前的输出/运行，状态机转为 idle 时，Runtime 自动检测到 Pending Events，将工具结果整理好喂给 Agent，触发下一个 Turn 自动运行。

## 模块五：Agent Runtime 架构对比

### Q1
Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

### A1
1. **机制不同点**：
   **OpenAI-compatible Function Calling (结构化 JSON)**：
     工具调用是协议级的，与文本输出分离。LLM 通过返回一个特殊的 JSON schema 结构来声明调用的工具与参数（如 `tool_calls` 列表）。Runtime 截获该 JSON 结构并执行，最后通过独立的 `tool` 角色消息返回结果。
   **Claude Code 工具调用 (流式 XML 标签)**：
     工具调用直接嵌入在普通的 text stream 中。LLM 在自由输出文本的过程中，只要写出特定 XML 标签（如 `<bash_call>npm test</bash_call>`），Runtime 的流式解析器（Streaming XML Parser）就会实时捕捉到该闭合标签，立刻中断文本流，开始执行对应的工具。
2. **优缺点对比**：
   **OpenAI 结构化 JSON 模式**：
     **优点**：规范度极高，易于通过 JSON Schema 做强类型校验；对模型尺寸要求较低（7B 等小模型也可以通过微调稳定输出 JSON）；不容易出现解析歧义。
     **缺点**：灵活性差，无法在一个交互轮次中做到“边想、边用工具、用完继续解释”。而且 Schema 的传递往往占用较大的 System Context，且难以直接支持流式边解析边运行（必须等 JSON 输出完整后才能解析运行）。
  **Claude Code 文本 XML 标签模式**：
     **优点**：极致的灵活性与流畅性。模型可以极其自然地进行 CoT 推理，并在长文本中无缝穿插调用多个不同的工具；支持流式边写边跑（比如只输出到 `</bash_call>` 就直接执行，无需等后面的思考内容输出完）；提示词开销更小。
     **缺点**：对模型的 Instruction-following 能力要求苛刻，弱模型经常写错标签或漏掉闭合标签；Runtime 端需要编写复杂的、容错率高的流式正则/XML解析器，开发维护难度大。
**我的实操过程中的感受**：对强模型（至少百B参数以上两者相差不大，对中等参数模型我感觉XML更稳定更灵活，对小模型都不行，虽然说是有语法错误风险，但是大模型写代码其实还是比较强的，只要代码不是特别长，35B模型json输入输出有时候会有问题，要做prompt工程约束。

### Q2
Openhands的状态机设计有什么优缺？更优雅的实现方式是怎么样的？

### A2
1. **OpenHands状态机的优缺点**：
   **优点**：
     **高度解耦与追溯性**：每一个 Action 和 Observation 均以结构化 Event 的形式在中心总线上发布，极易记录、重放（Replay）及做审计日志。
     **多模块协同方便**：各种 Monitor、GUI、Agent 都可以监听同一个 Event Stream，做到界面与内核的解耦。
   **缺点**：
     **开发链路冗长（Boilerplate 过多）**：即使是极小的功能改动，也需要定义新 Event、实现 Event Handler、维护复杂的状态转换树，写起代码来非常沉重。
     **并发控制困难**：当面对用户输入、异步工具回调、超时重试等多源并发事件时，很容易发生状态竞态条件（Race Conditions），导致状态机死锁或跳转错误，调试成本极高。
2. **更优雅的实现方式**：
   **基于 Python 异步生成器（Async Generator）的结构化并发模式**：
     摒弃大而全的全局显式状态机，将 Agent 的主循环写成一个线性的协程生成器。Agent 内部通过 `yield Action` 输出指令，Runtime 接收到后调度工具执行，并将结果通过 `generator.send(Observation)` 塞回。整个代码逻辑是纯线性的，极易阅读、调试，并且完美适配协程的上下文取消与生命周期管理。
   **轻量DAG编排（类似于 LangGraph）**：
     将不同执行阶段（如 Planning, Execution, Verification）定义为图的节点，状态转换通过“条件边”和简单的“局部 Context 变量”流转。这种设计能够同时具备状态机的分支决策能力，又避免了中心化大状态机带来的样板代码堆积。
